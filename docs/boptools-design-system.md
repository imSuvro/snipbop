# Boptools UI/UX Protocol

Boptools products are small, serious tools with a playful pulse. The design goal is not decoration; it is immediate confidence. A visitor should understand the task in seconds, feel the product has taste, and finish without wondering where the important control went.

## Product Promise

- One tool, one visible job, one primary workbench.
- The first screen must contain the actual utility, not a marketing detour.
- Delight should come from material, rhythm, and response, not from clutter.
- Privacy and trust claims must be plain, close to the action, and never inflated.
- Help is progressive: short hints in the tool, deeper guidance one click away.

## Visual Direction

Boptools uses an uncommon "desk instrument" style: ink type, paper surfaces, acid highlight, leaf utility accents, and coral warnings. The palette should feel handmade but precise.

- Backgrounds use quiet texture, hairline grids, and soft radial light.
- Panels are few and purposeful. Avoid stacking cards inside cards.
- Corners stay mostly tight: 8px for controls, 16px for large work surfaces.
- Big type is reserved for the opening brand/tool promise.
- Motion should be short, useful, and calm. Respect reduced-motion settings.

## Interaction Rules

- Primary actions are large enough to hit comfortably and placed in the workbench.
- Secondary actions sit nearby but visually quieter.
- Advanced choices appear only after the image is ready or when the user asks.
- Every drag, paste, export, and error state must tell the user what happened.
- Keyboard users get visible focus, logical tab order, and no hidden traps.

## Content Rules

- Lead with verbs: paste, choose, export, replace.
- Keep explanations short near the tool; move device/browser detail to help pages.
- Preserve SEO phrases where they matter, but never bury the utility.
- Use exact privacy language when promised, including "No upload to SnipBop."

## Reuse Checklist

Every future Boptools app should ship with:

- A workbench-first home screen.
- A one-page help route for edge cases.
- A small FAQ only when questions are real.
- Configurable canonical domain metadata.
- Light/dark theme support with system default.
- WCAG-aware contrast, focus, and pointer target sizing.
